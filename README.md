# JS Cartographer

JS Cartographer is an AI-powered JavaScript deobfuscator, code humanizer, and semantic mapper. It combines Abstract Syntax Tree (AST) transpilation recovery (via Wakaru) with Large Language Models (LLMs) and offline heuristic models to reconstruct minified, obfuscated, or bundled JavaScript code into human-readable, semantically annotated source files.

Beyond deobfuscation, JS Cartographer reconstructs module dependency and call graphs, recovers API surfaces (including fetch/axios sinks), exports valid OpenAPI 3.0.0 specifications, and hosts an interactive local dashboard.

---

## Architecture Overview

JS Cartographer leverages a **Map-Reduce In-Memory Pipeline** designed for ESM compatibility, strict TypeScript types, and high-performance AST manipulation:

1. **Unbundling & Recovery (Optional / Map Phase Start):** Raw Webpack/CommonJS bundles can be pre-processed to split modules.
2. **Wakaru AST Sanitization:** Rebuilds async/await state machines, ES6 classes, template literals, Yoda conditions, and splits sequence/merged variables.
3. **Identifier Filtering:** Optimization layer that filters out standard global scopes, DOM bindings, and pre-existing descriptive variables to reduce LLM tokens by 40-60%.
4. **LLM Naming:** Traverses scopes in descending order of block size to contextually rename remaining obfuscated variables using structured JSON mappings.
5. **Static Analysis & Local Extraction:** Extracts imports/exports, defined function mappings, call relationships, and network boundaries.
6. **Graph Reduction (Reduce Phase):** Uses `enhanced-resolve` to canonicalize import paths across modules and stitch files into global module, call, and API surface graphs.
7. **Local Dashboard:** Hosts a high-performance Express server and local web app showing the code visualizer.

## 🛠️ Setup Guide

### Environment Configuration
JS Cartographer uses a `.env` file (loaded via `dotenv`) to manage external API keys, server configurations, and LLM providers. Before running the project, configure your environment:

1. **Copy the example configuration:**
   ```bash
   cp .env.example .env
   ```
2. **Configure your keys and variables in `.env`:**
   * **`PORT`** *(Optional)*: Port for the visualizer dashboard server (defaults to `3000`). CLI `--port` option takes precedence.
   * **`DEFAULT_OUTPUT_DIR`** *(Optional)*: Default output directory (defaults to `./dist-output`). CLI `--output` option takes precedence.
   * **`LLM_PROVIDER`** *(Optional)*: Naming engine to use. Options are:
     * `heuristic` (default): Local rule-based offline naming (does not require any API keys/network).
     * `gemini`: Google Gemini API. Requires `GEMINI_API_KEY`.
     * `openai`: OpenAI API. Requires `OPENAI_API_KEY`.
     * `openrouter`: OpenRouter API. Requires `OPENROUTER_API_KEY`.
     * `local`: Local model execution (GGUF). Requires `LOCAL_LLM_PATH`.
   * **`LLM_MODEL`** *(Optional)*: Model name override (e.g. `gemini-1.5-flash` or `gpt-4o`).
   * **`GEMINI_API_KEY`** / **`OPENAI_API_KEY`** / **`OPENROUTER_API_KEY`**: API keys required for cloud LLM providers.
   * **`LOCAL_LLM_PATH`** *(Optional)*: Path to local GGUF model for local mode.

### Prerequisites
* **Node.js**: v18.0.0 or higher
* **npm**: v9.0.0 or higher

### Installation
1. Clone the repository and navigate to the project directory:
   ```bash
   git clone <repository-url>
   cd cartographer
   ```
2. Install all production and development dependencies:
   ```bash
   npm install
   ```

### Building the Project
JS Cartographer is built in strict TypeScript. Compile the source code to the `dist/` directory before executing commands:
```bash
   npm run build
   ```

## 💻 CLI Usage

The CLI interface is powered by Commander and is executable via `node dist/cli/run.js`.

### 1. Running the Deobfuscation Pipeline (`run`)
This command runs the Map-Reduce pipeline against a target folder of JavaScript files, processes them, generates graph outputs, and starts the visualizer dashboard.

```bash
node dist/cli/run.js run <input_directory> [options]
```

#### Options:
* `-o, --output <dir>`: Directory where the cleaned output code and metadata should be written. (Default: `./dist-output`)
* `--no-sanitizer`: Disables Wakaru structural sanitization and Yoda-condition flipping.
* `--no-heuristic-naming`: Disables constant simplification and heuristic name normalization.
* `-p, --port <number>`: Port on which to spin up the local visualizer dashboard. (Default: `3000`)

#### Example:
To deobfuscate a folder of minified scripts located in `./minified-src` and write the humanized output to `./readable-src` while launching the web server on port `8080`:
```bash
node dist/cli/run.js run ./minified-src --output ./readable-src --port 8080
```

### 2. Querying the Call Graph (`graph`)
This command queries the aggregated call graph generated by a previous pipeline run. It allows you to output ASCII call trees or save Mermaid flowchart specifications.

```bash
node dist/cli/run.js graph <output_directory> [options]
```

#### Options:
* `-e, --entry <function_id>`: The specific function ID to trace from as the root of the tree (e.g. `src/main.js:init`). *Required for ASCII tree output.*
* `-d, --depth <number>`: Maximum traversal depth to trace. (Default: `5`)
* `-f, --format <type>`: Output format: `tree` (ASCII tree in stdout) or `mermaid` (saves a Mermaid `.mermaid` file in the directory). (Default: `tree`)

#### Examples:
* **ASCII Call Tree**: Trace the calls made by `init` in `index.js` up to 3 levels deep:
  ```bash
  node dist/cli/run.js graph ./readable-src --entry "index.js:init" --depth 3
  ```
* **Mermaid Export**: Export the entire project-wide call graph to a Mermaid flowchart definition:
  ```bash
  node dist/cli/run.js graph ./readable-src --format mermaid
  ```

## 🌐 Local Explorer Dashboard (Webapp)

When you run the pipeline (`cartographer run`), it automatically launches an Express backend server at `http://0.0.0.0:<port>`.

### Features
* **Interactive Landing Page**: Displays server statistics and provides instant navigation.
* **REST APIs**: Exposes robust, sanitization-secure endpoints for external UI tools:
  * `GET /api/files`: Lists all deobfuscated files recursively.
  * `GET /api/file?path=<relative_path>`: Retrieves the humanized JavaScript source code of a specific file. Prevents directory traversal attacks via safe resolution checks.
  * `GET /api/graphs`: Serves a consolidated JSON payload containing `moduleGraph`, `callGraph`, and `apiSurface`.

### API Payload Outputs
The outputs saved in your output folder include:
* `module-graph.json`: Maps dependencies and records exports for each module.
* `call-graph.json`: Stores call hierarchies and edges mapped between functions.
* `api-surface.json`: Summarizes endpoint paths, request methods, and parameters.
* `openapi.json`: A standard, valid OpenAPI 3.0.0 specification file corresponding to your API surface.
