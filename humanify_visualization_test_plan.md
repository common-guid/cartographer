# Humanify Visualization Test Plan

This document outlines the testing strategy to validate the integration between the **HumanifyService**, the **PipelineOrchestrator**, and the **Web App Visualization** (Explorer). The goal is to ensure that code processed and renamed by the LLM (via Humanify) correctly flows through the entire pipeline and is accurately displayed in the web dashboard.

## 1. Pipeline Execution Validation
**Objective:** Confirm that the CLI correctly orchestrates the humanification process and outputs the results.

- [ ] **Data Flow:** Run `bin/cartographer run <test-directory>`.
- [ ] **File Output:** Verify that the output directory (e.g., `dist-output/`) contains the processed JS files.
- [ ] **Content Verification:** Open the processed JS files and verify that obfuscated variables/functions have been renamed with semantic names by Humanify.
- [ ] **Metadata Verification:** Ensure that `.metadata.json` sidecar files are generated and correctly reflect the structure of the *new* (renamed) code.

## 2. Reducer Phase & Graph Generation
**Objective:** Verify that the `ReducerService` correctly interprets the humanified code when building dependency graphs.

- [ ] **Module Graph:** Inspect `dist-output/module-graph.json`. Verify that the imports/exports reflect any structural or naming changes (if applicable).
- [ ] **Call Graph:** Inspect `dist-output/call-graph.json`. Confirm that the nodes in the call graph use the **new semantic names** generated by Humanify (e.g., a node should be named `initDatabase` instead of `_0x1a2b`).
- [ ] **Line Mappings:** Ensure that the line numbers stored in the call graph correctly map to the line numbers in the *humanified* output file, not the original raw source.

## 3. Backend API Validation
**Objective:** Ensure the Express server (`src/explorer/server.ts`) correctly serves the updated files and graphs.

- [ ] **`/api/files`**: Request the endpoint and confirm it returns the list of processed `.js` files, excluding sidecars like `.metadata.json` or graph JSONs.
- [ ] **`/api/graphs`**: Request the endpoint and ensure it serves the newly generated `moduleGraph`, `callGraph`, and `apiSurface` JSON objects containing the humanified names.
- [ ] **`/api/file?path=...`**: Request a specific file and verify the server returns the humanified source code content. Ensure it successfully blocks path traversal attempts.

## 4. Frontend Visualization Validation
**Objective:** Confirm that the React Web App (Sigma.js + Monaco Editor) correctly displays the humanified names and code.

- [ ] **Graph Rendering (Sigma.js):**
    - Load the dashboard and switch to the **Call Graph**.
    - Verify that node labels display the humanified function names.
- [ ] **Node Interaction & Context Panel:**
    - Click a node in the Call Graph.
    - Verify that the bottom-left context panel displays the humanified name, file path, and correct line number.
- [ ] **Code Editor (Monaco):**
    - When clicking a file in the sidebar or a node in the graph, ensure the Monaco editor loads the humanified source code.
    - Verify that the code displayed in the editor exactly matches the output generated by Humanify.
    - If clicking a specific function node, optionally verify if the editor can scroll to or highlight the corresponding line number (if implemented).

## 5. End-to-End Test Run
**Objective:** Perform a full manual or automated test on a realistic obfuscated sample.

1. Prepare a test fixture directory containing obfuscated code (e.g., `fixtures/obfuscated-sample`).
2. Run `npm run start run fixtures/obfuscated-sample`.
3. Open `http://localhost:3000` in the browser.
4. Search for known obfuscated functions in the original code, and verify they appear under their new semantic names in the visualization.
5. Click through the visualization to read the humanified code, checking for syntax errors introduced during the pipeline reintegration.
