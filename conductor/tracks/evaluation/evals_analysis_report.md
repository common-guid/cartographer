# Review: JS Cartographer Evaluation Strategies
> generated by Fable

## Overall Verdict

The document is **directionally sound** — precision/recall against ground truth, execution testing, and LLM-as-judge are the right family of techniques. However, it has **three significant problems**:

1. **A factual error about how filtering works** that invalidates part of Section 2's premise.
2. **It assumes measurement infrastructure that doesn't exist** (accurate token counts, failure counters, a ground-truth corpus, an alignment mechanism between minified and original identifiers).
3. **It underestimates the hardest problem**: none of the strategies explain *how to align* Cartographer's outputs (renamed identifiers, deobfuscated files) with ground-truth artifacts (original names, original files). That alignment layer is a major portion of the effort and is unmentioned.

Below is a section-by-section assessment.

---

## 1. Renaming Evaluation

### 1.1 Execution Testing

**Will it work?** Partially.

- The premise assumes target bundles come with tests. In practice, most obfuscated/minified artifacts do not.
- Silent fallback can inflate pass rates: if renaming fails and original code is returned, tests can still pass while rename quality is effectively zero.

**Recommendation:** pair pass/fail with:
- unchanged-output rate
- rename coverage metrics
- structural/alpha-equivalence checks

Execution tests are useful, but not sufficient as a primary metric.

### 1.2 Deterministic Semantic Accuracy

**Will it work?** Yes, but only with proper alignment.

- String-only metrics without robust identifier alignment can be misleading.
- Levenshtein is weak for synonym semantics.

**Recommendation:** prioritize:
- exact match rate
- token-level precision/recall/F1
- optional embedding similarity

### 1.3 LLM-as-a-Judge

**Will it work?** Yes.

This is appropriate for synonym-level semantic comparison that deterministic string metrics miss.

**Requirements for trustworthiness:**
- fixed prompt + fixed judge model version
- calibration against a human-labeled sample
- report central tendency and spread (not only mean)

### 1.4 Provider / Model Performance Metrics

**Will it work?** Latency/error/fallback: yes. Token/cost: partially.

- Token counts may be estimated in parts of current instrumentation.
- Cost reporting must distinguish estimated vs provider-native usage.

**Recommendation:** always report:
- latency p50/p95
- generation error rate
- fallback rate
- estimated tokens/cost
- native tokens/cost (when available)

## 2. Filtering Evaluation

### 2.1 Token/Size Reduction

**Will it work?** Yes.

`FilterStats`/`appCodeRatio`-style metrics are practical and valuable.

**Caveat:** estimated token savings should not be presented as exact billing-equivalent token counts.

### 2.2 Precision and Recall

**Will it work?** Yes, but the proposal's risk framing needs correction.

Filtering in the current pipeline primarily scopes what is sent to renaming and then reintegrates into the full AST. This means the key failure mode is often reduced rename quality/coverage or wasted token spend, not direct final-code deletion.

**Recommended primary metrics:**
- app recall (critical)
- app precision
- rename coverage on app-labeled nodes
- wasted token spend on false-app classifications

---

## 3. Call Graph & Data Flow Evaluation

### 3.1 Ground Truth Graph Generation

**Will it work?** Yes, with the right baseline strategy.

Ground truth should be fixture-based and reproducible, with expected artifacts in Cartographer-compatible schema.

### 3.2 Graph Similarity Metrics

**Will it work?** Yes.

Node/edge precision-recall and cross-file resolution metrics are appropriate.

**Caveat:** robust alignment is required (source maps, path normalization, structural signatures), not solely transformed identifier text.

### 3.3 Taint / API Surface Tracking

**Will it work?** Yes, and likely best near-term ROI.

Endpoint method + URL pattern manifests are straightforward to validate deterministically.

## Cross-Cutting Recommendations

1. **Dataset first**: build and version a benchmark corpus (toolchains, minifiers, source maps, manifests).
2. **Tiered evaluation model**:
   - Tier 0: deterministic CI checks
   - Tier 1: scheduled deep deterministic checks
   - Tier 2: on-demand LLM/provider benchmarking
3. **Track fallback masking explicitly** via unchanged-output + fallback rates.
4. **Label token/cost provenance** (estimated vs native).
5. **Capture visual evidence** (graph/runtime screenshots) for manual verification checkpoints.
