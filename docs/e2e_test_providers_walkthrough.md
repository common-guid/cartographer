# Walkthrough: Running Multi-Provider E2E Tests in JS Cartographer

Welcome! This guide is designed to take you from a freshly cloned repository all the way to running the multi-provider end-to-end (E2E) comparison test suite and analyzing the generated comparison reports. 

Whether you are running the system offline (using mocked LLM responses) or live (against production AI APIs), this tutorial covers all setup steps in the precise order they must be executed.

---

## 🧭 System Overview

The E2E Multi-Provider Comparison feature evaluates and compares the deobfuscation outputs of three different backends:
1. **`heuristic`**: A local, offline AST-based renamer that resolves minified variable names based on code syntax.
2. **`gemini`**: An AI-powered deobfuscator that uses the Google Gemini API to analyze context and rename variables humanely.
3. **`openrouter`**: An AI-powered deobfuscator that executes deobfuscation via the OpenRouter API.

The test suite runs these providers against a minified code fixture, saves the individual outputs, calculates execution duration and file size metrics, generates side-by-side snippets, and runs `git diff` to show the difference in renaming quality.

---

## 🛠️ Step-by-Step Setup and Execution

### Step 1: Clone the Repository
Begin by cloning the repository and navigating into the project directory:
```bash
git clone <repository-url> cartographer
cd cartographer
```

### Step 2: Install Node.js Dependencies
Ensure you have **Node.js (v18+)** installed. Install the dependencies:
```bash
npm install
```
> [!NOTE]
> During installation, a `postinstall` script runs `node scripts/install-humanify.js`. This automatically checks for and downloads necessary binaries (such as Wakaru or local Puppeteer utilities) required for certain stages of the deobfuscation pipeline.

### Step 3: Build the Project
JS Cartographer is written in TypeScript, while the CLI orchestrator and test scripts run on Node.js using compiled JavaScript. You **must** build the project before running the tests:
```bash
npm run build
```
This command:
1. Compiles all TypeScript files in `src/` to `dist/`.
2. Compiles the local visual dashboard client inside `src/explorer/client/`.

> [!IMPORTANT]
> Because the test runner `tests/run-provider-comparison.js` dynamically imports `dist/cli/orchestrator.js`, skipping the build step will result in a `Cannot find module` error. Always run `npm run build` after making modifications to TypeScript code.

### Step 4: Configure the Environment (`.env`)
The project contains a template configuration file named `.env.example`. Copy this file to create your active `.env` file:
```bash
cp .env.example .env
```
Open the newly created `.env` file in your preferred text editor. Locate the **LLM Provider Credentials** section and populate the keys for the providers you want to test:
```env
# Google Gemini API Key (Get one at: https://aistudio.google.com/)
GEMINI_API_KEY=AIzaSy...

# OpenRouter API Key (Get one at: https://openrouter.ai/)
OPENROUTER_API_KEY=sk-or-...
```

## 🚀 Running the E2E Test Command

You can run the E2E test in two distinct modes: **Mock Mode** (local-only, offline, free) or **Live Mode** (connects to real APIs, uses tokens).

### Mode A: Mock Mode (Offline-Friendly)
If you want to verify that the E2E pipeline, dynamic environmental overrides, and file output mechanisms work without invoking live APIs (saving time and money), run:
```bash
MOCK_LLM=true npm run test:providers
```

#### What happens under the hood?
1. The script verifies that the test fixture `fixtures/webpack-hello-world/dist/bundle.js` is present.
2. The output directory `dist-output-e2e/` is cleared.
3. For **heuristic**, it runs the real AST renamer locally.
4. For **gemini** and **openrouter**, it bypasses the network and uses `tests/helpers/mock-humanify.js` to simulate LLM renaming via a local regex-replace of obfuscated identifiers (e.g. renaming `_0x1a2b` to `mock_gemini_1a2b`).
5. It compiles metrics, runs `git diff --no-index` comparison baselines, and generates `dist-output-e2e/comparison_report.md`.

### Mode B: Live Mode (Real API Execution)
To benchmark actual AI model performance, run the command without the mock variable:
```bash
npm run test:providers
```

#### Graceful Skipping
If you do not define an API key (e.g. `OPENROUTER_API_KEY`) in your `.env` file, the script will output a warning and gracefully proceed with the remaining providers instead of crashing:
```text
⚠️ Skipping openrouter: No API key found
```

## 📊 Reading and Interpreting the Results

All files generated by the E2E process are saved in the `dist-output-e2e/` folder. The primary entry point for review is the **`dist-output-e2e/comparison_report.md`** file.

The generated report contains three sections:

### 1. Execution Summary Table
An overview of the execution status of all configured backends.
| Provider | Status | Execution Time | Output Size | Model |
|----------|--------|----------------|-------------|-------|
| heuristic| ✅ Run  | 1.10s          | 19074 B     | Gemini 3.5 Flash (Low) |
| gemini   | ✅ Run  | 0.00s          | 9450 B      | Gemini 3.5 Flash (Low) |
| openrouter| ⚠️ Skip | N/A            | N/A         | N/A   |

* **Status Codes:**
  * `✅ Run`: The provider completed deobfuscation successfully.
  * `⚠️ Skip`: The provider was skipped due to a missing API key.
  * `❌ Err`: The pipeline encountered an exception, displaying the failure message in the table.
* **Execution Time:** Compares parsing speeds. Heuristic and mock modes are near-instantaneous; live LLMs usually range from 5 to 20 seconds.
* **Output Size:** Tracks differences in the generated file size. Large differences can signify different formatting rules (e.g. minified vs formatted).

### 2. Output Snippets
Shows the first 30 lines of code emitted by each successful provider.
* **Why this is useful:** Allows quick visual check of coding conventions, variable/function naming consistency, and style formatting between the providers.

### 3. Unified Diffs (`vs heuristic baseline`)
The report embeds a unified `git diff` comparing the outputs of `gemini` and `openrouter` against the local `heuristic` baseline.
* **Why this is useful:** It lets you see exactly how the AI chose to rename specific obfuscated variables relative to static rules.
* **How to read it:**
  * Red lines starting with `-` represent the variable/function names or formatting generated by the `heuristic` renamer.
  * Green lines starting with `+` represent the refactored and humanized code names chosen by the AI provider (`gemini` or `openrouter`).
